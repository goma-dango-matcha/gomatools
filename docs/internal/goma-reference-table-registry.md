# ゴマ早見表 候補台帳 Version1.5

最終更新：2026-07-22

> このドキュメントはゴマツール開発・運営専用の内部資料です。
>
> 利用者向けの公開資料ではありません。

## 目的

ゴマ早見表シリーズとして追加を検討する候補を一元管理する。

候補を思いつきだけですぐ実装せず、検索需要、利用者価値、ブランド適合性、保守性を評価し、優先順位を決めるための管理資料として使用する。

---

## この文書の役割

- 本台帳：候補ごとの概要、評価、優先度、状態、関連情報を管理する
- [ゴマ早見表ロードマップ](goma-reference-table-roadmap.md)：シリーズ単位の検討順序とフェーズを管理する
- [ゴマ早見表 ブランド仕様書](../goma-reference-table-spec.md)：シリーズの役割とブランド方針を管理する
- [ゴマ早見表 共通実装仕様書](../goma-reference-table-implementation-spec.md)：すべての早見表へ適用する実装・確認基準を管理する
- 個別仕様書：採用した候補固有のデータ、情報源、URL、SEO、更新方法を定義する

本台帳への登録は、仕様化・実装・公開の決定を意味しない。

---

## 管理方針

- 候補は実装前に必ず本台帳へ登録する
- 十分な価値を確認できた候補だけを個別仕様書の作成対象とする
- 未調査の評価を推測で記入しない
- 公開後も削除せず、公開済み資産として履歴を残す
- 候補のフェーズや検討順序はロードマップで管理し、本台帳へ重複記載しない

---

## 管理項目

各候補について、次を管理する。

- 名称
- カテゴリ
- 概要
- 検索需要
- 保守コスト
- ブランド適合性
- ゴマ知識展開
- 関連ツール
- 優先度
- 状態
- 備考

---

## カテゴリ

原則として、次のカテゴリを使用する。

- 日付・時間
- 年齢
- 誕生日
- 生活
- 仕事
- 学習
- 健康
- その他

既存分類では管理できない候補が増えた場合に限り、重複しない名称でカテゴリ追加を検討する。

## 評価基準

### 検索需要

| 評価 | 基準 |
| --- | --- |
| ★★★★★ | 非常に高い |
| ★★★★☆ | 高い |
| ★★★☆☆ | 普通 |
| ★★☆☆☆ | 低い |
| ★☆☆☆☆ | 限定的 |

検索需要は、確認できるデータを根拠に評価する。データ未確認の場合は「未評価」とする。

### 保守コスト

| 評価 | 基準 |
| --- | --- |
| ★★★★★ | ほぼ保守不要 |
| ★★★★☆ | 年1回程度の確認 |
| ★★★☆☆ | 定期確認が必要 |
| ★★☆☆☆ | 更新頻度が高い |
| ★☆☆☆☆ | 保守負荷が大きい |

星が多いほど、長期運営で維持しやすいことを表す。

### ブランド適合性

| 評価 | 基準 |
| --- | --- |
| ★★★★★ | ブランドに非常によく合う |
| ★★★★☆ | 相性が良い |
| ★★★☆☆ | 問題なし |
| ★★☆☆☆ | 慎重に判断する |
| ★☆☆☆☆ | ブランドと合わない |

### ゴマ知識展開

| 評価 | 基準 |
| --- | --- |
| ★★★★★ | 複数の記事へ展開できる |
| ★★★★☆ | 記事を書きやすい |
| ★★★☆☆ | 記事化できる |
| ★★☆☆☆ | 展開が限定的 |
| ★☆☆☆☆ | 記事化が難しい |

未公開記事名やURLは推測せず、展開可能性だけを評価する。

### 関連ツール

既存ツールとの回遊性を確認し、自然に案内できる公開中ツールだけを記録する。

関連ツール名は[`tool-registry.md`](tool-registry.md)で実在と公開状態を確認する。カテゴリ一致だけで判断しない。

| --- | --- |

## 採用基準

次を満たす候補を優先する。

- 検索需要または明確な利用者需要が確認できる
- 保守コストが低い
- スマートフォンで一覧性を保てる
- 入力せず利用できる
- ブランドコンセプト「毎日に、一歩便利。」と一致する
- ゴマ知識へ展開できる
- 関連ツールとの自然な回遊をつくれる
- 信頼できる情報源と更新方法を確保できる

---

## 保留基準

次に該当する場合は、解決方法を確認できるまで保留する。

- 法改正が多い
- 料金改定が多い
- 外部APIへ依存する
- 自治体、地域、契約による差異が大きい
- 頻繁な更新が必要になる
- ブランドとの整合性が低い

---

## 見送り基準

次に該当し、構成の見直しでも解決できない場合は見送る。

- 既存ツールや既存コンテンツと役割が重複する
- 検索需要・利用者需要が低い
- 保守負荷が高過ぎる
- ブランド価値を高めない
- 利用者メリットが少ない
- 信頼できる情報を継続して提供できない

---

## 運用

1. 新しい候補を本台帳へ登録し、状態を「未検討」とする
2. 根拠を確認しながら各項目を評価する
3. 優先度と状態を更新する
4. 採用候補だけ個別仕様書を作成し、状態を「仕様作成」とする
5. 実装・レビューの進行に合わせて状態を更新する
6. 公開後は状態を「公開済み」とし、台帳から削除しない

定期的に検索需要、ブランド適合性、保守性を再評価する。評価根拠が変わった場合は、備考へ理由を記録する。

---

## 更新ルール

次の場合に更新する。

- 新しい候補を追加した
- 評価結果または優先度を決定・変更した
- 状態が変わった
- 関連ツールを確認・変更した
- 保留・見送り理由が変わった
- 公開後の保守条件が変わった

---

## 完了条件

- ゴマ早見表候補を一元管理できる
- 候補ごとの評価、優先度、状態を追跡できる
- 実装判断の根拠として利用できる
- 公開後も履歴を維持できる
- ロードマップ、ブランド仕様書、共通実装仕様書と役割が重複していない
